What Are Web Forms?

A web form is a simple form of interaction between a user and a web app. They allow the user to send data either to a web server or used on the client-side to update the page in some meaningful way. Examples of this would be updating a list, showing a UI feature, adding a product to a basket or joining a mailing list

Form Controls

Form controls or widgets are used to build the form and provide interactivity or purpose. They are generally created using the <form> elemtn and can provide the user with various ways of inputting the data. These include:

  • single line text fields
  • multi line text fields
  • radio buttons
  • checkboxes
  • dropdown boxes

Form Validation

Form validation refers to the fact that the form controls can be programmed to enforce or require specific formats or values when input. They are generally paired with text labels to provide semantic meaning as well as express the purpose of the form or it's various parts

The <form> element

All forms will start with the <form> element. It is a container element that formally defines the form and semantically marks it as such. It has some inherrent attributes as well as supporting specific attributes to tell the form how to behave and how to process the information included in it. Most of the attributes are optional but it is generally considered goofd practice to include at the very least the action and method attributes:

  • Action: Defines the URL where the collected data will be sent once submitted
  • Method: defines the type of metho to send the data (usually either POST or GET)

Here is the example code for this first stage:

<form action="/my-handling-form-page" method="post">...</form>

The <label>, <input> and <textarea> elements

There will be 3 very simple data entry areas with a connected <label>:

  • The input field for the name is a single-line text field
  • The input field for the email is an input of type email. This is a single-line text area that only accepts email addresses. An error will be thrown to the screen should a domain name not be included
  • The input filed for the message that will be sent is a <textarea> which is a multiline text field

With the addition of these form controls (or widgets), the code now looks like this:

<form action="/my-handling-form-page" method="post">...</form>
  <ul>
    <li>
      <label for="name">Name:</label>
      <input type="text" id="name" name="user_name"/>
    </li>
    <li>
      <label for="mail">Email:</label>
      <input type="email" id="mail" name="user_email"/>
    </li>
    <li>
      <label for="msg">Message:</label>
      <textarea id="msg" name="user_message"></textarea>
    </li>
  <ul>
</form>

The <li> elements are used to format the form and code and make styling simpler. An explicit label is used for each form control (widget) for usability and accessibility. This is added to by the use of a for attribute on each <label> element which is provided with an id that matches the for attribute in each label. This is to associate each form control with each label

This is becuase associating a label with the form control/widget enables a whole bunch of features such as mouse, trackpad and touch device activation. A user using any of those different input methods can simply click it to activate it as well as providing semantic meaning for users making use of screen readers

With the <input> element, the type attribute is the most important to add. It defines the way in which the <input> element will be presented and behave

  • In the example, the first input has a type of text. This will display a single-lline text field that accepts any text input
  • The second input has a type of email. Again, this is a single-line text area but this particular area will require a well-formed email address. It turns the filed into an "intelligent" field that perfroms a little bit of validiation to ensure that it includes an @ symbol
  • The third input does not exist but does include <textarea>

The <button> element

The final aspect needed to make the form functional is a button that allows the input data to be sent to the designated server/location. This is achieved by including a <button> element

The example code for this form looks like this:

<li class="button">
  <button type="submit">Send your message</button>
</li>

This is placed just above the closing </ul> tag in the HTML

The <button> element also needs a type attribute with one of three different values:

  • submit: sends the data included in the form to the web page that has been defined inside the action attribute on the <form> element
  • reset: resets all widgets to their default value
  • button does nothing unless fucntionality is added through JavaScript. Allows you to define your own functionality

Sending the form data to the web server

The <form> element defines where and how to senf the data. These two aspects are defined by the action and method attributes

A name attribute is added to each form control element (input element). Whatever is written in that attribute is attached to the data entered into the related field as metadata. This is the name part of name/value pairs. Whatever is written into the field is attached to the name attribute. When this is sent to the server, the script that runs on the server to handle sent data evaluates the data included in each name and then organises or palces it into the relevant database entry

So in the example form, the names of the name value pairs are:

  1. user_name
  2. user_email
  3. user_message

On the server/database, there will be a section that corresponds to each of the above names. The form is sent (pairing the name and the value of each into a name/value pair), evaluated and then organised into the ocrrect section
